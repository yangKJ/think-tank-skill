# think-tank Protocol

协议版本：`0.1.0`

## 1. 协议目标

think-tank 协议定义一套平台无关的协作分析流程，用于把复杂问题从输入请求推进到多角色分析、讨论审议、结论汇总和行动建议。

协议必须保持稳定。平台适配和场景模式可以扩展执行细节，但不能改写本协议的核心阶段和输出结构。

## 2. 触发条件

当任务满足任一条件时，应考虑触发 think-tank：

- 需要多渠道信息收集
- 需要多个专业角色分别判断
- 存在明显不确定性、争议或取舍
- 用户要求讨论、评审、研究、策略、审议、委员会、专家组或观点碰撞
- 单一回答容易遗漏风险、反例或执行细节

当任务非常简单、无需多角色或无信息收集价值时，不应强行触发完整流程。

## 3. 输入契约

think-tank 输入应被解析为：

- `task`：用户要解决的问题
- `mode`：场景模式，可显式指定，也可由协议推断
- `context`：已知背景、文件、资料、项目状态
- `constraints`：时间、范围、工具、平台、语言、输出格式等限制
- `success_criteria`：用户如何判断结果有用
- `evidence_policy`：是否允许联网、是否只能使用本地仓库、是否需要引用来源
- `output_preference`：报告、计划、审查清单、决策建议或执行步骤

## 4. mode 选择

默认支持：

- `research`：以信息收集、证据组织和研究结论为中心
- `council`：以多角色讨论、交叉质询和审议为中心
- `review`：以发现问题、验证质量和提出修复建议为中心
- `strategy`：以目标、取舍、路线和执行优先级为中心

若用户显式指定 mode，优先使用用户指定。若未指定，应根据任务目标推断，并在输出中说明采用的 mode。

## 5. 角色选择机制

角色选择必须服务任务，而不是固定凑人数。

基础角色池：

- `collector`：收集信息、证据、上下文和约束
- `domain_expert`：从领域专业角度分析
- `skeptic`：寻找漏洞、反例、风险和过度自信
- `builder`：关注落地路径、实现成本和执行顺序
- `synthesizer`：整合结论、分歧和行动建议

不同 mode 可以定义默认角色组合，但必须保留按任务动态调整的空间。

## 6. 流程阶段

标准流程为：

1. `intake`：解析任务、目标、约束和成功标准
2. `mode_selection`：选择或确认场景模式
3. `role_planning`：选择角色、职责和分析重点
4. `collection`：收集资料、证据、上下文和现有实现
5. `independent_analysis`：角色独立分析，避免过早共识
6. `deliberation`：讨论、质询、反驳、修正观点
7. `synthesis`：归纳共识、分歧、风险和结论
8. `recommendation`：输出行动建议、优先级和下一步
9. `quality_check`：检查协议完整性、证据边界和输出质量

平台可以压缩低风险任务的阶段，但不能跳过 `synthesis` 和 `quality_check`。

## 7. 终止条件

满足以下条件时可以终止：

- 已回答用户核心问题
- 关键分歧已列明并解释
- 行动建议已具备可执行性
- 证据不足或工具受限的部分已明确标注
- 输出满足对应 mode 的质量门禁

不应因为角色之间达成表面一致就提前终止。

## 8. 输出结构

标准输出应包含：

- `结论`：直接回答或核心判断
- `依据`：关键证据、上下文或推理来源
- `角色观点`：不同角色的主要判断
- `分歧与风险`：未解决争议、风险和不确定性
- `行动建议`：具体下一步，按优先级排列
- `边界`：未验证、不能确认或需要后续补充的部分

轻量任务可以合并栏目，但不能丢失结论、风险和行动建议。

## 9. 质量标准

think-tank 输出必须：

- 不把猜测写成事实
- 不把 mock、demo、tracking 路径说成已验证真实能力
- 不抹平角色分歧
- 不输出只有概念、没有行动的结论
- 不把平台适配细节混入协议层判断
- 能说明为什么选择当前 mode 和角色组合
