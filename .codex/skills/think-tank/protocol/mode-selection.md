# Mode Selection

本文件定义 think-tank 如何从用户请求选择场景模式。

mode 是协议层的执行策略，不是平台功能开关。平台可以用不同方式执行 mode，但不能改变 mode 语义。

## 可用 mode

- `research`：研究、信息收集、证据整理和研究结论
- `council`：多角色讨论、观点碰撞、审议和决策
- `review`：审查、验收、质量判断和修复建议
- `strategy`：产品、架构、路线、优先级和长期取舍

## 选择优先级

1. 用户显式指定 mode 时，优先使用用户指定。
2. 用户描述包含明确场景词时，按场景词推断。
3. 用户请求包含多个目标时，选择最能决定输出形态的 mode。
4. 无法可靠判断时，使用 `council` 作为通用审议模式，并说明原因。

## 旧 Claude Code 触发词归并

旧 research think-tank 和 agent-council 的触发词应归并为 mode 选择语义，而不是继续作为平台专属规则存在。

| 旧触发 | 新 mode | 说明 |
|--------|---------|------|
| `深度调研` | `research` | 多渠道、多角色研究 |
| `协作调研` | `research` | 多角色并行收集后汇总 |
| `团队调研` | `research` | 研究团队式收集和分析 |
| `多人调研` | `research` | 多角色研究 |
| `深入分析` | `research` 或 `strategy` | 以证据为主用 research，以路线为主用 strategy |
| `开个会` | `council` | 多角色审议 |
| `开会讨论` | `council` | 多角色讨论 |
| `讨论一下` | `council` | 通用审议 |
| `简单讨论` | `council` | 轻量 council |
| `快速讨论` | `council` | 轻量 council |
| `思维碰撞` | `council` | 强制保留分歧和反驳 |
| `评审一下` | `review` | 以问题发现和验收为中心 |

普通 `调研`、`搜索`、`查一下` 默认不触发完整 think-tank，除非任务同时要求多角色、证据冲突处理、讨论或最终决策。

## 场景词

### research

触发词或意图：

- 研究
- 调研
- 查资料
- 信息收集
- 对比资料
- 来源
- 证据
- 现状分析

输出倾向：

- 研究报告
- 证据表
- 结论和置信度
- 后续验证路径

### council

触发词或意图：

- 讨论
- 审议
- 专家组
- 委员会
- 观点碰撞
- 正反意见
- 多角色
- 是否应该
- 帮我判断

输出倾向：

- 共识结论
- 分歧点
- 少数意见
- 风险
- 决策建议

### review

触发词或意图：

- 审查
- review
- 验收
- 找问题
- 评估产出
- 看是否正确
- 稳定性
- 缺陷
- 测试缺口

输出倾向：

- 按严重程度排序的问题
- 文件或证据定位
- 修复建议
- 验收结论
- 残余风险

### strategy

触发词或意图：

- 策略
- 路线
- 规划
- 架构方向
- 产品定位
- 优先级
- 阶段计划
- 长期演进

输出倾向：

- 战略判断
- 路线图
- 优先级
- 阶段目标
- 决策点

## 冲突处理

当请求同时匹配多个 mode：

- 先研究再决策：使用 `research`，并在结尾给出 council 风格决策建议。
- 先验收再修复：使用 `review`，并在结尾给出 builder 风格修复计划。
- 先定位再规划：使用 `strategy`，但必须包含 collector 的上下文整理。
- 明确要求多角色争论：优先使用 `council`。

## 输出声明

每次使用 think-tank 时，应在内部或输出中记录：

```yaml
selected_mode: ""
selection_reason: ""
roles: []
```

轻量输出可以不展示完整字段，但执行过程必须遵守该选择。
